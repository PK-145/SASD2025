# [Kent Beck – ทำไม Code Review ถึงพัง (และจะแก้อย่างไร)](https://www.youtube.com/watch?v=LW3xRESuh4M)

## แนวคิดหลัก
ปัญหาหลักของ Code Review แบบดั้งเดิม **ไม่ใช่เรื่องคุณภาพโค้ด เครื่องมือ หรือความสามารถของนักพัฒนา**  
แต่คือ **การออกแบบกระบวนการ (process design)** ที่บังคับให้ Code Review เป็นทั้ง  
**การบล็อกงาน (blocking)** และ **ต้องทำพร้อมกัน (synchronous)**  
ซึ่งทำให้ต้นทุนสูง คาดเดาไม่ได้ และไม่สอดคล้องกับธรรมชาติของงานเชิงความรู้

## ปัญหาเชิงโครงสร้างของ Code Review
- **Blocking**: ผู้เขียนโค้ดต้องรอการอนุมัติก่อนจะไปต่อหรือ merge ได้
- **Synchronous**: การรีวิวขึ้นอยู่กับเวลาว่างและลำดับความสำคัญของคนอื่น
- **Incentive ไม่สอดคล้องกัน**:
  - ผู้เขียนโค้ดต้องการ: รีวิวเร็ว และมีคุณภาพ
  - ผู้รีวิวต้องการ: ใช้แรงน้อย ทำเมื่อสะดวก
- ผลลัพธ์: latency สูง, ควบคุมไม่ได้, และเกิด coordination cost

## Code Review จริงๆ แล้วทำอะไรบ้าง
Code Review เป็นคำครอบคลุมงานหลายประเภท:
- การหาบั๊ก
- การประเมินสถาปัตยกรรม
- ความสะอาดและมาตรฐานของโค้ด
- การทำความเข้าใจว่าการเปลี่ยนแปลงใหม่เชื่อมกับระบบเดิมอย่างไร

## Insight จากงานวิจัย
จากงานวิจัยที่ Kent Beck อ้างถึง:
- ส่วนที่ใช้เวลามากที่สุดของ Code Review **ไม่ใช่การหาบั๊ก**
- และ **ไม่ใช่การตรวจคุณภาพโค้ด**
- แต่คือ **การพยายามทำความเข้าใจการเปลี่ยนแปลงของโค้ด**
- ต้นทุนหลักคือ cognitive load ของผู้รีวิว

## แนวคิด “Code Feed”
Kent Beck เสนอแนวคิดเชิงทดลองชื่อว่า **Code Feed**:
- ได้แรงบันดาลใจจาก news feed ของโซเชียลมีเดีย
- เปลี่ยนจาก “โพสต์” เป็น “การเปลี่ยนแปลงของระบบ”
- แสดงเฉพาะสิ่งที่เกี่ยวข้องและน่าสนใจต่อแต่ละคน
- ให้ situational awareness แบบต่อเนื่อง โดยไม่บล็อกการทำงาน

### ผลลัพธ์ของ Code Feed
- ทีมมีความเข้าใจร่วมกันว่าระบบกำลังเปลี่ยนอะไร
- ลดการพึ่งพา approval-based review
- ทำงานร่วมกันแบบ asynchronous ได้จริง

## ทางเลือกที่ดีที่สุดในทางปฏิบัติ
เมื่อไม่มีระบบรีวิวแบบไม่บล็อก:
- ต้องลดต้นทุนการทำความเข้าใจโค้ดให้มากที่สุด
- ทำให้โค้ดอ่านแล้วเข้าใจได้ทันที
- รองรับการทำงานร่วมกันแบบ asynchronous

## ข้อสรุปสำคัญ
ยิ่งโค้ดเข้าใจง่ายเท่าไร  
ต้นทุนของ Code Review แบบบล็อกงานก็ยิ่งลดลง
